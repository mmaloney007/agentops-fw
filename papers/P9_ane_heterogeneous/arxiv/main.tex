% ABSTRACT
% ============================================================================
\begin{abstract}
We trained an SLO-aware agent entirely on a laptop.

Papers~I--IV in this series established that small language models face a capacity threshold during GRPO fine-tuning.  Below a critical size, models lose the ability to produce valid structured output and never recover.  Those experiments required datacenter GPUs.  This paper asks whether the same training can run on commodity hardware, using Apple's Neural Engine for inference and Metal GPU for gradients.

We evaluate a hybrid ANE+MLX architecture on four model configurations.  The Apple Neural Engine delivers 55~tok/s at an estimated 3W for Qwen2.5-0.5B (roughly 10$\times$ the power efficiency of Metal GPU inference).  But raw inference speed is not the bottleneck.  On-device GRPO hits the same capacity threshold we saw on server hardware.  Qwen2.5-0.5B collapses after 4 training steps, Qwen3.5-0.8B after 3, and Qwen3.5-2B after 6.  All lose JSON validity completely with no recovery.

The hybrid architecture changes this.  We route rollout generation to the ANE and gradient computation to Metal, synchronizing weights between engines after each step.  The result: 100\% JSON validity across all 5 training steps, a 43\%/57\% ANE/Metal compute split, and per-step times of 4.7--10.9 seconds.

On-device GRPO training is viable---but only through heterogeneous compute that exploits each engine's strength.
\end{abstract}

% 1. INTRODUCTION

\section{Introduction}

Every GRPO training run in Papers~I--IV required a datacenter GPU.  An NVIDIA RTX~4090 with 24GB of VRAM, consuming 350W under load, running for hours per model.  The results were worth it: we established capacity thresholds, reward stability boundaries, conditions under which small models can learn structured output.  But the hardware requirements excluded anyone without a research cluster or a high-end workstation.

This paper asks a different question: can the same training happen on a laptop?

This isn't an abstract question.  Privacy-sensitive deployments (healthcare, legal, financial) can't send training data to cloud GPUs.  Edge robotics needs on-device adaptation.  And the economics work: a MacBook Pro costs \$2,500 once, while GPU cloud instances cost \$1--4 per hour indefinitely.

Apple Silicon offers an unusual opportunity.  Every M-series chip contains two compute engines.  The Neural Engine (ANE) is a 16-core matrix accelerator optimized for inference at $\sim$3W.  The Metal GPU is a general-purpose compute unit capable of automatic differentiation through MLX~\cite{hannun2023mlx}.  Neither alone suffices.  The ANE can't compute gradients.  The Metal GPU wastes power on inference that the ANE handles at 10$\times$ better efficiency.

We built a hybrid architecture that exploits both.  The ANE generates rollouts (the forward passes that produce candidate responses during GRPO training).  Metal computes the policy gradients and updates LoRA weights.  A lightweight synchronization step transfers the updated weights back to the ANE's CoreML representation.

Four results stand out.  First, Qwen2.5-0.5B hits 54.8~tok/s with a 650ms time-to-first-token on the Neural Engine, producing 100\% valid JSON on live tasks at roughly 10$\times$ the power efficiency of Metal GPU inference.  Second, pure MLX training on Metal replicates the collapse patterns from Paper~II~\cite{maloney2025capacity}: Qwen2.5-0.5B holds JSON validity for 4 steps then collapses permanently, Qwen3.5-2B holds for 6 steps.  The threshold is a property of the model, not the hardware.  Third, our ANE+MLX architecture achieves 100\% JSON validity across all 5 training steps, with rewards stable at 1.84$\pm$0.01 and a 43\%/57\% ANE/Metal compute split.  Fourth, weight synchronization through unified memory costs under 1ms per step, making the hybrid overhead negligible.

Section~\ref{sec:background} covers the ANE and GRPO background.  Sections~\ref{sec:architecture}--\ref{sec:setup} describe the hybrid architecture and experimental setup.  Section~\ref{sec:results} presents results.  Section~\ref{sec:discussion} discusses implications for on-device RL.

\section{Background}
\label{sec:background}

\subsection{Apple Neural Engine}

The Apple Neural Engine (ANE) is a fixed-function matrix accelerator present in every M-series Apple Silicon chip.  The M-series ANE contains 16 cores capable of up to 15.8 trillion operations per second, operating at approximately 3W~\cite{apple2022m2max}.  Unlike the Metal GPU, the ANE executes only CoreML-compiled models; it cannot run arbitrary compute graphs or backward passes.

CoreML compilation imposes constraints.  Models must be traced through \texttt{coremltools}~\cite{apple2023coremltools}, which converts PyTorch operations to the Model Intermediate Language (MIL).  Several operations common in transformer inference (\texttt{gather} for embeddings, \texttt{one\_hot}, and \texttt{repeat\_interleave}) are not natively supported on the ANE execution path, falling back to CPU or GPU~\cite{anemll2024}.  Getting a model to run \emph{purely} on the ANE requires operation-level rewrites that are beyond the scope of this work.

\subsection{MLX on Metal GPU}

MLX~\cite{hannun2023mlx} is Apple's array computation framework for Metal GPU, designed as a NumPy-compatible alternative to PyTorch for Apple Silicon.  What matters for this work: MLX supports automatic differentiation and LoRA fine-tuning, so it can handle on-device gradient computation.  MLX models run on the Metal GPU at about 30W, drawing from the unified memory pool shared with the ANE and CPU.

\subsection{GRPO}

Group Relative Policy Optimization~\cite{grpo} generates $G$ candidate responses per prompt, scores them against a reward function, and updates the policy using relative advantage within each group.  Unlike PPO~\cite{schulman2017ppo}, GRPO eliminates the critic network.  That makes it more memory-efficient, which matters on-device where VRAM is limited.

Papers~I--IV~\cite{maloney2025deploymentgap,maloney2025capacity,maloney2025agentslobench,maloney2025dynamics} established that GRPO fine-tuning of small language models for structured output faces a \emph{capacity threshold}.  Models below a critical size lose JSON validity during training and never recover.  Paper~II placed this threshold between 1B and 3B parameters on server-class hardware.  This paper tests whether the same threshold holds on-device.

\subsection{ANE Inference Provider}

The ANE inference provider wraps a CoreML-compiled model behind the same interface used by the MLX provider.  A prompt is tokenized, fed through the CoreML model via Apple's \texttt{MLModel} API, and decoded autoregressively.  The provider handles structured output by constraining generation to valid JSON tokens, the same schema-guided decoding used in Papers~I and~III~\cite{maloney2025deploymentgap,maloney2025agentslobench}.

CoreML compilation proceeds through \texttt{coremltools} tracing.  We trace only the decode path ($S=1$), since GRPO rollouts generate one token at a time.  The KV cache is managed through CoreML state tensors, registered as \texttt{register\_buffer} in the PyTorch wrapper and modified in-place during forward passes.

\subsection{MLX GRPO Trainer}

The MLX trainer implements GRPO with LoRA adaptation on Metal GPU.

\begin{itemize}[leftmargin=*]
\item \textbf{LoRA configuration:}  Rank 8, targeting 8 layers, applied to attention projections.  This keeps the trainable parameter count under 2M for all models tested.
\item \textbf{Gradient clipping:}  Maximum norm of 1.0.  Paper~II identified gradient clipping as essential RL infrastructure: without it, reward variance in early steps produces NaN gradients that corrupt the entire model.  We confirm this finding on-device.
\item \textbf{Group size:}  $G=4$ for hybrid training, $G=2$ for pure MLX baselines.  Larger groups improve advantage estimation but increase rollout time proportionally.
\end{itemize}

\subsection{Weight Synchronization}

After each gradient step, updated LoRA weights must transfer from the MLX representation to the CoreML model.  The idea is simple: extract weight deltas, apply them to the CoreML model's state dictionary.  The implementation is less simple, requiring careful handling of data types and memory layouts.

In practice, weight synchronization takes under 1ms per step (Table~\ref{tab:hybrid_timing}).  Both engines share Apple Silicon's unified memory, so synchronization is a pointer operation rather than a memory copy.  On a discrete GPU system with separate memory, this step would involve PCIe transfers.

\section{Experimental Setup}
\label{sec:setup}

\subsection{Hardware}

All experiments run on a MacBook Pro with Apple M-series Silicon, 36GB unified memory, 16-core Neural Engine, and 18-core Metal GPU.  No external GPU, no cloud compute, no network calls during training.

\subsection{Models}

We evaluate four model configurations spanning two model families:

All models use 4-bit quantization (MLX format).  The Qwen2.5-0.5B hybrid configuration uses CoreML for ANE rollouts and MLX for gradient computation.  The three pure-MLX configurations serve as baselines, running both inference and training on Metal GPU.

\subsection{Task and Reward}

Training uses CLINC intent classification~\cite{larson2019clinc150} with 500~records.  Each prompt asks the model to classify a user utterance into one of 150 intents, returning a JSON object with \texttt{intent} and \texttt{confidence} fields.  The reward function combines:

\begin{itemize}[leftmargin=*]
\item \textbf{JSON validity} (0.0 or 0.3):  Whether the response parses as valid JSON matching the required schema.
\item \textbf{Intent accuracy} (0.0 or 0.5):  Whether the predicted intent matches the gold label.
\item \textbf{Confidence calibration} (0.0--0.2):  Bonus for well-calibrated confidence scores.
\item \textbf{Format penalty} ($-$0.1):  Deducted for thinking-mode token leakage (\texttt{<think>} tags in output).
\end{itemize}

Maximum reward per sample is 1.0.  The $\beta=0.3$ KL penalty coefficient prevents the policy from diverging too far from the reference model.

\subsection{On-Device GRPO Stability}
\label{sec:stability}

The capacity threshold from Paper~II replicates on consumer hardware, step for step.

We ran GRPO training on three pure-MLX configurations as baselines before testing the hybrid.  Every one of them eventually collapses, losing JSON validity completely.

Table~\ref{tab:stability} reports the collapse dynamics for each configuration.  The pattern is consistent: models produce valid JSON for a few steps, then abruptly transition to 0\% validity with no recovery.

\begin{table}[h]
\centering
\caption{GRPO stability comparison across four configurations. ``Valid Steps'' counts consecutive steps with 100\% JSON validity from the start. ``Total Steps'' is the full training run length.}
\label{tab:stability}
\begin{tabular}{llccc}
\toprule
\textbf{Model} & \textbf{Trainer} & \textbf{Valid Steps} & \textbf{Total Steps} & \textbf{Last-$N$ Valid} \\
\midrule
Qwen2.5-0.5B & Hybrid (ANE+MLX) & \textbf{5/5} & 5 & \textbf{100\%} \\
Qwen3.5-2B & MLX (Metal) & 6/16 & 16 & 0\% \\
Qwen2.5-0.5B & MLX (Metal) & 4/25 & 25 & 0\% \\
Qwen3.5-0.8B & MLX (Metal) & 3/16 & 16 & 0\% \\
\bottomrule
\end{tabular}
\end{table}

Look at the step-by-step dynamics.  Qwen2.5-0.5B on pure MLX starts strong: steps~0--3 produce valid JSON with rewards between 0.46 and 1.96.  At step~4, reward drops to $-$0.028 and validity hits 0\%.  It never recovers across the remaining 21 steps, oscillating between $-$0.034 and $-$0.028.  Not low rewards.  Zero validity.

Qwen3.5-2B shows the threshold most clearly.  It holds 100\% validity and stable rewards ($\sim$1.92) for 6 steps, the longest of any pure configuration.  At step~6, it collapses to $-$0.068 reward and 0\% validity.  The 2B model's extra capacity buys more stable steps but can't prevent the eventual collapse.

Qwen3.5-0.8B collapses fastest, at step~3.  Its initial reward is strong (1.95 at step~0) but the smaller capacity can't sustain learning.  There's also a confound: Qwen3.5 models in thinking mode leak \texttt{<think>} tokens into the output, eating the generation budget and triggering format penalties.

\textbf{Gradient clipping prevents NaN but not collapse.}  All configurations use gradient clipping with max norm~1.0.  Without it, the 0.5B model produces NaN gradients at step~2, corrupting all weights permanently.  With clipping, the model survives longer but still collapses at the capacity threshold.  Clipping prevents catastrophic failure.  It can't overcome the underlying capacity limitation.

\textbf{ANE rollouts account for 43\% of compute time.}  Averaging 3,592ms per step, the ANE generates 4 candidate responses per prompt.  This is competitive with Metal GPU inference: MLX generates at 64~tok/s for the 0.8B model but the ANE generates at 55~tok/s for 0.5B while drawing $\sim$10$\times$ less power.

\textbf{Metal gradient computation accounts for 57\% of compute time.}  The MLX backward pass averages 4,703ms, with high variance (1,939--6,171ms).  The variance reflects the variable-length responses generated by the ANE; longer responses require more memory and computation in the gradient step.

Weight synchronization is effectively free.  Unified memory means LoRA weight transfer takes under 1ms.  On a system with discrete GPU and separate memory, PCIe transfers could dominate the training loop.

The reward signal is the real differentiator.  The hybrid architecture maintains rewards between 1.836 and 1.860 across all 5 steps (mean 1.846, std 0.01).  The pure MLX baselines crash from $\sim$1.9 to $\sim$$-$0.03 upon collapse.

\section{Discussion}
\label{sec:discussion}

\subsection{On-Device Tensions}

On-device GRPO training runs into constraints that don't exist on server hardware.

\textbf{Capacity vs.\ stability.}  Larger models resist collapse longer (6 steps for 2B vs.\ 3 steps for 0.8B) but need more memory and compute per step.  On a 36GB unified memory system, 4-bit 2B models eat about 1.5GB.  That's manageable, but it leaves little headroom for activations during gradient computation.

\textbf{Power vs.\ throughput.}  The ANE delivers competitive throughput at 3W but can't compute gradients.  Metal GPU can compute gradients but draws 30W.  A 5-minute training run on Metal drains about 2.5Wh of battery, negligible.  But a 500-step run at 8.3 seconds per step takes 69 minutes at 30W, consuming 34.5Wh---nearly a third of a typical MacBook Pro battery.

\textbf{Split vs.\ overhead.}  The hybrid architecture adds complexity: two model representations, a synchronization protocol, coordination logic.  The overhead is currently negligible ($<$1ms sync), but scaling to larger models or more frequent synchronization could change the picture.

\subsection{When ANE Makes Sense}

The ANE isn't universally better than Metal GPU inference.  MLX hits 64.1~tok/s for Qwen3.5-0.8B versus 54.8~tok/s for Qwen2.5-0.5B on ANE.  The GPU is faster in raw throughput for same-generation models.  The ANE advantage is power efficiency: when you need thousands of rollout generations during training, running them at 3W instead of 30W preserves battery and reduces thermal throttling.

Also, the ANE introduces fixed overhead.  CoreML model compilation takes 30--60 seconds, and the first inference call incurs a $\sim$650ms warmup.  For single-shot inference, Metal GPU via MLX is simpler and faster to start.  For training loops with hundreds of rollout generations, the per-inference power savings compound.

\subsection{Implications for the Paper Series}

The capacity threshold from Paper~II~\cite{maloney2025capacity} is not a datacenter artifact.  We see the same collapse dynamics on a laptop: valid JSON for a few steps, abrupt drop to 0\% validity, no recovery.  The threshold is a property of model capacity interacting with RL optimization, not of the hardware running it.

The hybrid architecture offers a partial solution that Paper~II didn't consider.  Decouple inference from gradient computation, and you can run rollouts on a larger frozen model while computing gradients on a smaller LoRA adapter.  That asymmetry maps naturally onto the ANE+Metal split.

\subsection{Thinking Mode as a Confound}

Qwen3.5 models trained with thinking mode enabled leak \texttt{<think>} tokens into structured output.  These tokens eat the 128-token generation budget and prevent complete JSON responses.  This is a training artifact, not a capacity limitation.  Two fixes exist: disable thinking mode during GRPO rollouts, or increase the generation budget to accommodate reasoning tokens.  Paper~V~\cite{maloney2025moe} flagged thinking-mode overhead as a deployment concern; here we see it directly interfering with training stability.

% 7. CONCLUSION
% ============================================================================
\section{Conclusion}
\label{sec:conclusion}

We trained an SLO-aware agent on a laptop using a hybrid architecture that routes inference to Apple's Neural Engine and gradient computation to Metal GPU.  The architecture maintains 100\% JSON validity across training steps where pure approaches collapse.

On-device GRPO training is viable, but not through brute force.  The same capacity threshold that limits small models on server hardware limits them on consumer hardware.  The ANE's power efficiency makes rollout generation practical at 3W, but it can't compute gradients.  Metal GPU handles gradients but wastes 10$\times$ the power on inference the ANE does better.

The hybrid splits compute 43\%/57\% between ANE and Metal, achieves per-step times of 4.7--10.9 seconds, and maintains stable rewards at 1.85$\pm$0.01.  Weight synchronization through unified memory takes under 1ms, making the architectural complexity nearly free.

The model that trains successfully on a laptop is not the one with the most parameters.  It is the one paired with an architecture that plays to each engine's strength.

\section{CoreML Conversion Notes}
\label{app:coreml}

Converting PyTorch transformers to CoreML for ANE execution is not plug-and-play.  Four issues came up:

\begin{itemize}[leftmargin=*]
\item \textbf{Tracing path:}  We trace only the $S=1$ decode path.  The prefill path ($S>1$) triggers different code paths in the attention mechanism that \texttt{coremltools} cannot trace through conditional branches.  In practice, GRPO rollouts are autoregressive, so only the decode path matters.

\item \textbf{KV cache as state:}  CoreML state tensors require the \texttt{name} attribute on \texttt{StateType}, not on \texttt{TensorType}.  We register KV cache tensors as \texttt{register\_buffer()} in the PyTorch wrapper and modify them in-place during the forward pass.

\item \textbf{GQA head expansion:}  The \texttt{repeat\_interleave} operation used for grouped-query attention is not supported on ANE.  We replace it with an \texttt{unsqueeze}$\,\to\,$\texttt{expand}$\,\to\,$\texttt{reshape} sequence that produces identical results through ANE-compatible operations.

\item \textbf{Dimension padding:}  ANE requires all tensor dimensions to be multiples of 32.  We pad trailing dimensions (e.g., conv cache dimension 3$\to$32) with zeros and slice the results after computation.
\end{itemize}

Not all models survive this process.  Of the 13 models in our Paper~I--IV test suite, only Qwen2.5-0.5B and Llama-3.2-1B compiled successfully.  The Qwen3.5 family failed due to unsupported operations in their attention implementation, so they run on MLX instead.
